# CareerLens：就活企業情報・締切検証システム

## プログラム構成

`app.py`はStreamlitの起動専用であり、実装は責務別に
`career_lens`パッケージへ分割している。

```text
app.py                         Streamlit起動ファイル
career_lens/
├── common.py                 設定、データモデル、JSON Schema、共通関数
├── auth.py                   アカウント認証、パスワードハッシュ、利用者コンテキスト
├── storage.py                SQLite、キャッシュ、確認履歴、締切蓄積
├── sources.py                情報源分類、Web検索、本文取得、機械候補抽出
├── ai_client.py              Gemini接続、再試行、代替モデル切替え
├── research.py               収集と検査を同時並行する逐次調査
├── ai_analysis.py            主抽出、独立検証、条件付き裁定
├── verification.py           原文、年度、コース、締切の整合性検証
├── scheduler.py              Z3日程作成、ICS・Googleカレンダー出力
└── ui.py                     Streamlit画面とアプリ全体の制御
tests/
├── test_accuracy.py          精度・安全性の回帰テスト
└── test_accounts.py          認証、個人データ分離、匿名集計の回帰テスト
```

## アカウントと共同確認

- 利用者ごとにアカウントを作成し、検索履歴、締切台帳、公式ドメイン判定、本人確認、Z3日程履歴を分離する。
- パスワードは平文で保存せず、利用者ごとに異なるソルトを付け、PBKDF2-HMAC-SHA256でハッシュ化する。
- ページ本文などの公開情報キャッシュは共有するが、AIのページ単位抽出キャッシュは利用者ごとに分離する。
- 他利用者の検索条件、履歴、氏名、アカウント名は表示もAI送信もしない。
- 同一企業・年度・募集区分・コース・締切について、「確認済み」と「誤情報」の人数だけを匿名集計する。
- 匿名集計はAIが根拠ページを探すための補助信号であり、原文より優先しない。
- 他利用者の確認だけでは、本人の確認済み状態やカレンダー候補へ昇格しない。

## ゲスト利用

- ログイン画面の「ゲストとして利用」から、アカウントを作成せずに開始できる。
- ゲストごとに推測困難なセッションIDを発行し、固定の共有ゲストIDは使用しない。
- 検索、複数AI検証、Python検証、Z3日程作成、ICS出力を利用できる。
- 締切確認と公式ドメイン確認はブラウザセッション内だけで保持する。
- 検索履歴、締切台帳、個人用AIキャッシュ、Z3日程履歴はDBへ保存しない。
- 公開Webページ本文のキャッシュのみ、通常利用者と共有して通信量を抑える。
- ブラウザセッションの終了、アプリの再起動又は「ゲスト利用を終了」で一時データは失われる。
- 永続履歴が必要な場合は、従来どおりアカウントを作成してログインする。

## 精度改善

- 未登録ドメインをURLや企業名だけで「公式」と判定しない。
- 情報源一覧で利用者が企業公式ドメインかを確認し、企業ごとに保存する。
- 締切の近くに対象年度、コース名、締切語、日付がある場合だけを機械確認済みとする。
- 検索結果の概要文しか取得できない場合は、候補として残すが機械確認済みにしない。
- 締切を0件しか取得できなかった場合は「合格」ではなく「締切未確認」と表示する。
- 「複数情報源の支持数」は、同じ日付があるだけでなく、企業名・対象年度・締切文脈を本文で確認できた情報源だけを数える。
- 不確実な候補は削除せず、情報源URLを開いて利用者が「確認済み」または「誤情報」に分類できる。

## 今回の中心的な変更

現在の課題であった「検索結果に含まれているのに、最終的な締切一覧から漏れる」問題を抑えるため、処理方式を一括処理から逐次処理へ変更した。

従来は、Web検索と本文取得がすべて終了した後に、複数ページをまとめてAIへ送信していた。現在の方式では、各情報源の本文を取得できた時点で、次の検査を開始する。

```text
複数の検索文を並列実行
        ↓ 検索結果が届いた順
候補ページの本文を並列取得
        ↓ 本文を取得できた順
Pythonによる日付・締切語の検査
        ＋
軽量AIによるページ単位の独立抽出
        ↓
締切候補を逐次表示・蓄積
        ↓
全情報源を統合した主抽出・独立検証・必要時のPro裁定
```

検索中にも、取得済みのページについて検査が進む。画面には検索数、本文取得数、逐次AI検査数、検出済み候補がリアルタイムに表示される。

## 収集漏れを抑える仕組み

- 通常の検索文に加え、応募締切、提出期限、エントリー期限、マイページ、PDF募集要項などを対象とする深掘り検索を追加した。
- インターンでは、コース、仕事体験、オープン・カンパニー、第1次・第2次締切などの表現も検索する。
- 標準モードでは、企業公式、マイナビ、ONE CAREERなどを検索した後、情報が不足する場合のみSNS・補助サイトを追加検索する。
- 網羅モードでは、主要情報源とSNS・補助サイトをすべて検索する。
- 最大36ページ（網羅モードは60ページ）の本文を取得する。
- 全ページをPythonで検査し、締切語の近くにある日付を候補として保持する。
- 上位ページは軽量AIが1ページずつ独立に読み、コースと締切の対応を抽出する。
- ページ単位で裏付けられた候補は、最後の全体抽出が見落とした場合でも「要確認候補」として補完する。
- 候補は実際の情報源本文で再検査し、候補だけを根拠として確定しない。

## 速度とAPI利用量の対策

- Web検索、本文取得、ページ単位AI検査を別々のスレッドプールで実行する。
- AIの逐次検査は同時2件、標準で最大14ページ、網羅モードで最大24ページに制限する。
- 全ページをAIへ送らず、締切語と日付の周辺だけを抜粋する。
- ページ単位AI検査の結果を72時間キャッシュする。
- 既存の検索結果・本文キャッシュも継続して利用する。
- Gemini 3.5 Flashが混雑している場合は、既存の再試行・フォールバック機構を利用する。

## 更新方法

1. 起動中のStreamlitを停止する。

```bash
control + C
```

2. 提出ZIPを展開し、展開したフォルダに移動する。

3. `.env.example`を`.env`として複製し、Gemini APIキーを設定する。

4. 次のファイルはGitや提出用ZIPに含めない。

- `.env`
- `.venv`
- `.venv_windows`
- `jobhunt.db`

5. 起動する。

```bash
source .venv/bin/activate
pip install -U -r requirements.txt
streamlit run app.py
```

## テスト

Gemini APIやWeb検索を呼び出さない回帰テストは、次のコマンドで実行できる。

```bash
python -m unittest discover -s tests -v
```

テストには、未確認ドメイン、別コースの締切、別年度、
検索概要のみ、締切0件、ゲストデータの非永続化とセッション分離などの異常系を含む。

`jobhunt.db`は初回起動時に自動生成される。既存のデータベースがある場合も、必要なテーブルが起動時に追加される。旧版の共有確認データは利用者別テーブルへ自動流用せず、別アカウントへ表示されない。

## 推奨設定

通常の利用では、次を推奨する。

- 検索方式：標準
- 収集と検査を同時並行で行う：オン
- 別モデルで抽出結果を再検証する：オン
- Proモデル：不一致・根拠不足がある場合だけ実行

情報収集の網羅性を優先する実験では、検索方式を「網羅」にする。網羅モードは検索数と取得ページ数が増えるため、標準モードより時間を要する。

## レポートで説明できる工夫

- 生産者・消費者型の逐次情報処理ワークフロー
- 収集と検証の同時並行化
- 情報取得順に依存しない候補の一元管理と重複排除
- Pythonによる一次検査とAIによる意味検査の役割分担
- 全体抽出の見落としをページ単位検査で補完する機構
- 情報不足時だけ探索範囲をSNSへ拡張する動的探索制御
- AI呼び出し数の上限、キャッシュ、関連箇所抜粋による計算量管理

## 注意

公開検索で取得できないマイページ内の情報、ログインが必要なページ、画像だけに記載された締切は、依然として取得できない場合がある。その場合は手動入力機能を利用する。

## Streamlit Community Cloudへの反映

ゲスト対応による新しい環境変数、DBテーブル及び依存ライブラリはない。
変更済みファイルを既存のGitHubリポジトリへ反映すると、Community Cloudが自動的に再デプロイする。

```bash
python -m unittest discover -s tests -v
git add career_lens/auth.py career_lens/storage.py career_lens/ui.py \
  tests/test_accounts.py README.md
git commit -m "Add non-persistent guest access"
git push origin main
```

デプロイ後は、シークレットウィンドウで公開URLを開き、次を確認する。

1. 「ゲストとして利用」ボタンから検索画面へ進める。
2. 検索、AI検証、Z3及びICS出力を利用できる。
3. ゲスト利用中で履歴が保持されない旨が画面に表示される。
4. 「ゲスト利用を終了」後に、ゲストの確認状態が残っていない。
5. 通常アカウントでは従来の履歴と確認状態が保持される。
